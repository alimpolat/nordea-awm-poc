§ 2 · What changed since 2024 LlamaIndex defaults

Seven shifts that matter for a 2026 demo

The existing project (LlamaIndex BasicRAG / SentenceWindow / AutoMerging + Chroma + BGE + BGE-reranker) was canonical for late 2023 / early 2024. Don't ship it unchanged to a Head-of-AI in 2026.

1
Naive RAG is dead-on-arrival as the headline

The story is now agentic RAG (the LLM plans its own retrieval) or contextual / hybrid retrieval (chunks know what they belong to). Submitting a naive top-K demo to a Head-of-AI in 2026 telegraphs you stopped reading in 2023. Hamel Husain's August 2025 "Beyond Naive RAG" mini-book is titled exactly for this reason.

2
Contextual Retrieval (Anthropic, Sept 2024) is now the chunking default

Prepend a 3-4 sentence LLM-generated summary of "what this chunk is about within its parent doc" before embedding. Reduces top-20 retrieval failures by 49% on its own, 67% with reranking. Cost amortized with prompt caching; on a 50-doc POC it's a one-time job.

3
Hybrid (dense + BM25) is table stakes, not advanced

Pure dense retrieval is now considered amateur for any corpus with named entities (tickers, ISINs, fund names, client codes). Recall jumps from ~0.72 (BM25) to ~0.91 (hybrid) in benchmarks.

4
GraphRAG hype has compressed

Microsoft's GraphRAG cost $33K to index large corpora; GraphRAG-Bench (ICLR'26) shows it frequently underperforms vanilla RAG on real-world tasks. HippoRAG and PathRAG are better on multi-hop reasoning, but none of them belong in an 8-day POC. The right answer for Nordea: gesture at graph-augmented retrieval as a future direction, not build it.

5
Long-context did not kill RAG

The "stuff 1M tokens into Gemini" pattern degrades >50% past 100K tokens in agentic settings (2026 research). Jerry Liu's framing — "you still need the right retrieval tools to access massive enterprise context" — is now consensus.

6
The framework wars are over and nobody won

LangChain is increasingly seen as legacy; LangGraph won the orchestration crown for stateful multi-step agents (Uber, Klarna, JPMorgan use it); LlamaIndex retained the data/ingestion crown; Pydantic AI is winning hearts on type-safety. The 2026 default for serious production is LlamaIndex for retrieval + LangGraph for orchestration. But for an 8-day POC, both are overkill — a thin FastAPI orchestrator beats either.

7
Eval has bifurcated

Ragas owns automated RAG metric scoring (faithfulness, context precision/recall, answer relevance). Phoenix (Arize) owns OpenTelemetry tracing. LangSmith owns the closed/managed UX. For a free, defensible, regulated-finance demo: Ragas + Phoenix-OSS, with the explicit caveat that production would add error-analysis-driven evals in Hamel's mode (binary pass/fail + human-aligned LLM judge).

§ 3 · Question-by-question findings

The eight calls in detail

Q1 — Framework: who won?

Verdict: Nobody. De-facto 2026 production stack is LlamaIndex (retrieval) + LangGraph (orchestration) + Pydantic AI (type-safe agent definitions). Many top teams ship custom.
  • LlamaIndex's Jerry Liu has explicitly repositioned the company as "more than a RAG framework — it is Agentic Document Processing," with a new pitch that data ingestion is 50%+ of an AI engineer's time and worth outsourcing to LlamaCloud (Jerry Liu, Aug 2024).
  • LangGraph has the production weight: Uber, Klarna, LinkedIn, JPMorgan, 400+ companies, mature checkpointing and HITL. But the steepest learning curve of any 2026 framework (2026 Decision Guide).
  • Hamel Husain explicitly refuses to recommend any framework. When teams ask, he redirects: "Can you show me how you're measuring if any of this actually works?" (Hamel Field Guide).
  • Jerry Liu's most recent (May 2026) framing: "Building RAG 2.0 is just making Claude Code running over your filesystem." Agentic loops over a virtualized filesystem with file-tools replacing static retrieval indices for code-style workloads (Jerry Liu, May 2026).
  • 2026 industry analysis: "Coding agents have disrupted the core value of RAG frameworks — when AI can generate custom pipelines on demand, pre-packaged abstractions lose their appeal, and MCP has standardized tool integration." (MindStudio).

For Nordea POC: Use LlamaIndex Workflows for ingestion/retrieval (mature, well-named for a banking audience, existing codebase uses it). Wrap with a 200-line async FastAPI orchestrator hand-rolled with Pydantic models for agent state. Do not introduce LangGraph in 8 days — hard to learn, harder to debug, zero demo value-add over async-await for the visible workflow.

Q2 — Embedding models: who wins for finance?

Verdict: gemini-embedding-001 (Vertex AI). Full stop.
  • Tops MTEB v2 English at 68.32, tested on "science, legal, finance, coding" (Google Developers Blog).
  • 3,072-dim with Matryoshka Representation Learning — truncate to 768 or 1536 without quality loss. Pick 768 for Qdrant memory budget.
  • Already on your GCP service account. Zero new signups. $0.15/MTok standard, $0.075/MTok batch. For 50 docs × ~10 chunks × ~500 tokens: ingestion ~$0.04.
  • Voyage 3 large is the only competitor that beats it on retrieval-focused metrics, but requires new paid API key → out per budget rule.
  • BGE-M3 local is the credible self-hosted fallback (BentoML 2026 OSS guide). For a Cloud Run deployment with no GPU, dragging BGE-M3 in means cold-start pain. Don't.
  • Cohere embed-v4 is competitive but, again, new paid signup.

Q3 — Vector DB for a POC

Verdict: Qdrant in Docker for the POC. Footnote: "production would use pgvector or Vertex Vector Search."
  • The 2026 vector DB market has consolidated: Qdrant, pgvector, LanceDB, Chroma, Weaviate are all production-viable; Pinecone increasingly seen as overpriced; Milvus enterprise-heavy (4xxi 2026, MarkTechPost 2026).
  • Chroma still wins for sub-1M-vector prototypes but is increasingly read as a "I built this in 2024" signal by senior reviewers. Avoid for a senior-leader demo.
  • Qdrant is the explicit 2026 recommendation for "building legal or financial AI with complex filtering" — JSON-payload filtering is the production differentiator the moment you filter by client / asset class / date / restricted-list status (CallSphere 2026).
  • pgvector 0.9 is the right long-term answer for Nordea (banking already runs Postgres). For 8 days, Qdrant has lower friction.
  • Vertex AI Vector Search is the GCP-native option and what production Nordea would use. But endpoint provisioning takes 30-60 min per change — iteration burn. Use Qdrant for POC, namedrop Vector Search as the production migration path.

For Nordea POC: docker run -p 6333:6333 qdrant/qdrant. Single collection, gemini-embedding-001 vectors at 768 dims, payload schema:

{
  "asset_class": str,
  "doc_type":    str,
  "as_of_date":  date,
  "client_id":   str | None,
  "source_uri":  str
}

The payload is what makes this defensible: in the chat-to-interrogate UI you can demo filter pushdown.

Q4 — Retrieval pipeline

Verdict: Hybrid (dense + BM25) + Anthropic Contextual Retrieval chunking + listwise LLM reranker. This is the 2026 baseline, not "advanced."
  • Contextual Retrieval is the single biggest free-lunch since 2023. Prepend a 3-4 sentence summary to each chunk before embedding ("This chunk is from Bergström Family Office Q1 review, discussing reallocation from Swedish small-caps to Gulf real estate, page 3."). Anthropic measured: 49% retrieval failure reduction alone; 67% with reranking (Anthropic Contextual Retrieval). Confirmed by independent benchmarks (+2.8pp Recall@5 dense, +2.2pp hybrid).
  • Hybrid is mandatory for finance corpora — tickers, ISINs, named clients, fund codes all favor BM25/sparse. Recall typically jumps from ~0.72 to ~0.91.
  • ColBERT / late interaction is sexy but not table stakes for a 50-doc POC. ColBERT's edge appears at scale and with long chunks; for 50 docs the operational complexity outweighs the gain. Verdict: "A dense model followed by a cross-encoder reranker recovers most of ColBERT's quality advantage with a more standard stack" (Codastra/Medium).
  • GraphRAG / HippoRAG / PathRAG: Skip for the POC. GraphRAG-Bench (ICLR'26) shows these "frequently underperform vanilla RAG on real-world tasks" while costing 10-100x more to index (arxiv 2506.05690). Gesture at it as "phase 2."
  • Reranking is non-negotiable. Either cross-encoder (bge-reranker-v2-m3 OSS leader, free) or LLM-as-reranker (Gemini 2.5 Flash listwise, ~$0.0001/query). For Nordea: use Gemini Flash listwise — fewer moving parts, no model download, native to stack.

For Nordea POC pipeline (5 stages):

  1. Query → Gemini 2.5 Flash query rewriter (resolves "Bergström" → "Bergström Family Office" + extracts intent)
  2. Parallel: (a) dense retrieval over Qdrant (gemini-embedding-001, top 30), (b) BM25 over same chunks (top 30)
  3. Reciprocal Rank Fusion merge → top 20
  4. Gemini 2.5 Flash listwise rerank → top 5
  5. Gemini 2.5 Pro answer synthesis with mandatory inline citation [doc_id:chunk_id]

Q5 — Agentic patterns

Verdict: Anthropic's orchestrator-subagent pattern, scaled down to 1 + 3 + 1.
  • Anthropic published the canonical 2025-2026 reference on multi-agent research systems: lead agent maintains plan in external memory, spawns parallel subagents with clear objectives + output formats + tool guidance, subagents work independently and return findings, lead synthesizes (Anthropic Engineering). Key finding: token usage explains 80% of performance variance — parallel subagents win because they distribute context, not because they're "smarter."
  • The 2026 enterprise consensus is the five patterns taxonomy: router, ReAct, plan-and-execute, multi-agent retrieval, self-RAG. ReAct dominates for single-agent flows; multi-agent retrieval shines on "heterogeneous sources" — exactly the Nordea case: docs + financial data + news (Heeya 2026 Agentic RAG Guide).
  • Jason Liu's framing (sunsetting his 567 Labs courses Feb 2026): the flywheel is ship → log → segment → specialize. He warns that in most RAG contexts, recall is the headache, not generation (Jason Liu, Jan 2025).
  • 2026 critical failure modes for multi-agent: excessive subagent spawning, duplicated work, context overflow. All have known mitigations (scaling rules, boundary descriptions, fresh subagent contexts).

For Nordea POC architecture:

query enters · planner fans out to 3 parallel subagents · findings converge at synthesizer · brief renders in the UI

The chat-to-interrogate uses a single ReAct agent with tools: retrieve_from_brief(), retrieve_from_corpus(), lookup_client_holdings(), web_grounding(). This is the demo's "wow" — and it's also defensible.

Q6 — Eval

Verdict: Ragas for metrics, Phoenix-OSS for tracing, 30-question hand-built eval set following Hamel's binary-pass-fail rule.
  • Ragas owns the RAG-specific metric suite (faithfulness, context precision, context recall, answer relevance). Open source. Lower cost than LLM-as-judge (Atlan 2026).
  • Phoenix (Arize) is observability-focused, OpenTelemetry-native. Pairs cleanly with Ragas. Open source, runs in-process (Braintrust 2026).
  • LangSmith is managed and requires a LangChain dependency. Skip — vendor lock-in story to a demo where you're explicitly trying to look vendor-neutral.
  • TruLens is fine but has lost mindshare to Ragas since mid-2025. No advantage for banking audience.
  • Hamel's overlay: binary pass/fail beats 1-5 scales; pair each pass/fail with a critique; validate LLM-judge alignment with human judgment before trusting metrics. For 8 days, aim ~80% alignment, not 95% (Hamel Field Guide).

For Nordea POC eval plan:

  1. Generate 30 synthetic questions from your 50-doc corpus using Gemini Pro (5 per chunk type, segmented by intent: lookup / multi-hop / quantitative / contextual / news).
  2. Hand-review and fix the 30. Mark expected source chunks.
  3. Run Ragas. Targets: context_recall > 0.85, faithfulness > 0.90, answer_relevance > 0.80. These are the numbers Thomas's team will benchmark against — and they're achievable in 8 days.
  4. Phoenix-trace every call. Export Phoenix screenshot for the concept memo.
  5. Build a 5-question "regression set" of hard cases (named-entity disambiguation, multi-doc synthesis, "I don't know" handling). These are the demo recordings.

Q7 — Gemini-specific

Verdict: Vertex AI Gemini 2.5 has four under-used capabilities that should shape the architecture.
  1. Native JSON / structured output (via response_schema). Pin all agent outputs to Pydantic schemas. Single biggest reliability lever. Planner's plan, each specialist's findings, the synthesizer's brief — all enforced schemas (Vertex Gemini 2.5 Pro docs).
  2. Grounding with Google Search. Returns citation URIs for any web-fetched fact. Only legally-defensible way to bring public news data into a Nordic Tier-1 bank brief — every fact has a source URI compliance can audit. Free tier 1,500 RPD shared, then $35/1k prompts. Free tier covers demo.
  3. Long context (1M tokens) — use sparingly. Recent research: agentic capabilities degrade >50% past 100K tokens (Long-context degradation research). Don't try to stuff the whole Bergström history. Use it for: "load the IPS document fully into the synthesizer's context, RAG everything else."
  4. Code execution + function calling. Gemini 2.5 Pro has native code execution. For portfolio math (weighted YTD return, VaR), use code execution rather than expecting the LLM to do arithmetic. AWM-relevant differentiator vs a generic chatbot.

Where Gemini loses vs Claude/GPT: Tool-use loops slightly clunkier than Claude's native tool blocks. Function-calling JSON sometimes drifts. Mitigation: enforce schemas, validate with Pydantic, retry on parse failure. Where Gemini wins: native Search grounding (Claude has nothing equivalent), native code execution, top MTEB embeddings in same SDK, 1M context, and the killer for Nordea — it's already approved on their cloud-provider list if they're a GCP shop.

Q8 — What's overhyped / NOT worth chasing in 8 days

Skip these. Hard.
  • GraphRAG. Microsoft's reference impl costs $33K to index large corpora; GraphRAG-Bench (ICLR'26) shows it frequently underperforms vanilla. Build a knowledge graph for Bergström family relationships? Beautiful in theory, 5 days of work, demo crash risk high. Skip.
  • HippoRAG / PathRAG / LightRAG. Same family. Research-grade. Don't ship.
  • ColBERT / late interaction first-stage retrieval. Operational complexity > demo value for 50 docs.
  • Fine-tuned embeddings. Jason Liu's "if you're not fine-tuning embeddings you're Blockbuster" line is correct for mature systems with logged interactions. Cold-start POC has no interaction data. Skip.
  • LangGraph. Best-in-class for production, but 2 of your 8 days burned on learning curve. Use FastAPI + async + Pydantic.
  • Multi-modal / chart understanding. Tempting for finance. Gemini Pro can do it, but adds ingestion complexity and demo brittleness. Save for v2.
  • Self-RAG / self-correction loops. Add latency, marginally improve quality, hard to demo. Skip.
  • Custom fine-tuned Gemini. No.
  • MCP server scaffolding. Trendy. Useless for a 50-doc closed-corpus POC.
  • "Evals as marketing." Don't show off Ragas dashboards. Show off the brief quality and the interrogation chat. Eval is the trust receipt, not the headline.

§ 5 · Anti-patterns and traps

Ten ways this POC can drift off-course

  1. Don't build a knowledge graph. GraphRAG hype has compressed; benchmarks show it loses to vanilla on real-world data; indexing cost is brutal. If asked "what about GraphRAG?" in the meeting: "We evaluated GraphRAG-Bench (ICLR'26). For closed-corpus advisor briefs, hybrid + contextual retrieval gives multi-hop wins at 1/100th the indexing cost. Graph is on the roadmap for cross-client pattern detection in phase 2."
  2. Don't ship without citations. Every fact in the brief must carry a [doc_id:chunk_id] or a Vertex grounding URI. Banking. Audit. Compliance. Non-negotiable.
  3. Don't use a 1-5 confidence scale. Hamel's research: binary pass/fail + critique > scales. Brief says "high confidence: Bergström rebalance complete" or "low confidence: rumored Q2 capital call — single source, unverified."
  4. Don't show off Ragas dashboards in the demo. Eval is in the appendix of the concept memo. Demo is the brief + chat.
  5. Don't pretend the POC will scale. Thomas knows it won't. Memo's "production path" section explicitly: "Qdrant → Vertex Vector Search; Cloud Run → GKE; add LangGraph for multi-day HITL workflows; fine-tune embeddings after 6 months of logged advisor interactions."
  6. Don't introduce a second cloud. No Pinecone. No Anthropic. No OpenAI. Single-vendor story is the trust argument.
  7. Don't use LangChain. Signals "started this in 2023." LlamaIndex Workflows reads more current.
  8. Don't demo on a flaky network. Bake corpus into the container, run locally in the meeting, screen-share Cloud Run separately as deployment proof.
  9. Don't let the synthesizer hallucinate quantities. Force quantitative answers through code_execution() tool calls. Show the code in the citation drawer.
  10. Don't oversell agentic-ness. "Multi-agent" is a buzzword Nordea has heard 50 times this year. Frame as: "Parallel specialists with bounded scope and structured handoffs — same pattern Anthropic shipped for their research product." Cite Anthropic explicitly. Borrowed credibility.
