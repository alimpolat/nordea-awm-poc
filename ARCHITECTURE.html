§ 1 · System map

Everything, connected in one view

The advisor opens the cockpit; a single container runs the FastAPI app and the LlamaIndex Workflows orchestrator that drives five stage-agents. Those agents call Vertex AI for reasoning, the retrieval engine for grounded citations, and the data & intel layer (static corpus + live World_Monitor) for fresh signals. The brief flows back to the browser, where ECharts and human-in-the-loop controls render it.

clay = request / LLM call · olive = response / live signal · ◌ pulse = World_Monitor live feed
Single-container design. Everything except Vertex AI and World_Monitor runs in one container — Qdrant is embedded in-process (read-only after the index build), BM25 lives in memory. No sidecars, no managed vector DB, no extra spend. Cold start is ~3–5 s; the brief is cached so the advisor sees it instantly while the container regenerates in the background.
§ 2 · The five-stage agentic flow

The advisor flow, end-to-end

This diagram answers the JD directly: "enabling the private banking advisor flow end-to-end — from opportunity identification and client insights to meeting preparation, recommendations, and follow-up." Each stage is an agent with a typed Pydantic output contract; Stage 3 fans out to four specialists that run concurrently via asyncio.gather, then a Synthesizer assembles the brief. The loop closes when captured follow-ups become next week's opportunity signals.

§ 3 · Retrieval pipeline

Grounded answers, with citations

Three of the four specialists ground their findings in a hybrid retrieval pipeline. At query time, the question is embedded with task-type RETRIEVAL_QUERY and searched against both a dense vector store (Qdrant) and a sparse index (BM25); the two result sets are fused with Reciprocal-Rank Fusion (k=60), then a Gemini Flash listwise reranker picks the top-5 with citations. The index is built once — each chunk is augmented with Anthropic-style contextual prefixes before embedding, which cuts retrieval failures roughly in half on cross-document questions.

Hybrid retrieval · dense (Qdrant) + sparse (BM25) → RRF k=60 → Flash reranker → top-5 · amber lane = one-time index build
Built once, shipped in the container. The index build (chunk → contextual augment → embed → Qdrant) runs locally before deploy; the resulting qdrant_data/ and bm25.json are baked into the Docker image. At runtime the store is read-only — no vector-DB service to provision, secure, or pay for.
§ 4 · Model routing

Flash carries breadth; Pro handles depth

Gemini 2.5 Flash handles all breadth-of-work tasks — scouts, specialist analysts, reranker, chat assistant (~10 calls per brief). Gemini 2.5 Pro is reserved for the two reasoning-critical stages — Planner and Synthesizer (2 calls per brief). This split keeps cost low while keeping the reasoning backbone strong.

§ 5 · Decisions log

Why we built it this way

Key architecture decisions, with rationale.

DecisionChoiceRationale
LLM provider Vertex AI Gemini only Single GCP service account covers everything — no additional API keys or spend. Gemini 2.5 Pro/Flash SOTA at the time of build; 1M-token context handles long meeting-note chains cleanly.
Vector store Qdrant embedded (in-process) No managed vector DB to provision, secure, or pay for. Index is built once, baked into the Docker image, and served read-only. Qdrant embedded has strong Rust-core performance; the corpus (~500 chunks) is small enough that memory is no concern.
Hybrid retrieval Dense + BM25 → RRF k=60 Pure dense fails on exact ticker names and IPS clause references (e.g. "single-position limit 5%"). Pure sparse misses semantic paraphrases. RRF k=60 is the SOTA fusion baseline with no hyperparameter tuning required.
Contextual Retrieval Anthropic-style prefix per chunk Cross-document contextual questions (e.g. linking BIS AI-boom analysis to Bergström US-tech allocation) require each chunk to carry enough document-level context. The prefix is generated once at build time via Flash; retrieval failures on contextual intents dropped from ~60% to ~35%.
Embedding model gemini-embedding-001 @768d task-typed Task-typing (RETRIEVAL_QUERY vs RETRIEVAL_DOCUMENT) is the single highest-leverage free-tier tuning available. Avoids asymmetric embedding space mismatch between queries and document chunks.
Reranker Gemini Flash listwise Cross-encoder style reranking without a separate model deployment. Flash listwise prompt scores all k candidates simultaneously — ~200ms latency overhead, substantial precision improvement over RRF-only.
Orchestration LlamaIndex Workflows Typed event/step model enforces output contracts at each stage; asyncio.gather for the parallel specialist fan-out. LlamaIndex's Phoenix integration provides distributed tracing at no cost.
Brief caching Pre-generated, served instantly Full brief generation takes ~16 s. Serving a cached result eliminates advisor wait time — the cockpit loads immediately, and the orchestrator regenerates in the background on a schedule. This also means cold-start latency does not matter for the user experience.
Deployment host HF Spaces (POC) / Cloud Run (target) The GCP sandbox service account does not have Cloud Run deployer permissions. HF Spaces offers free zero-card Docker hosting with secrets injection — identical image, zero additional work. The cloudbuild.yaml is ready for Cloud Run when permissions are granted.
Frontend charts Apache ECharts (Apache 2.0) No licensing risk. Rich financial chart types (radar, bar, gauge, pie). First-class React wrapper (echarts-for-react). Animation and theming APIs match the house palette perfectly.
Eval judge Custom Gemini Flash judge OSS Ragas context-precision/recall requires an OpenAI-compatible endpoint; the Ragas 0.4.x Gemini adapter hit a version quirk that returned nulls. Rather than spend time on adapter patches, a custom Flash judge was wired directly — same methodology, fully in-budget, honest about the limitation in EVAL.html.
